# BPF Map State Poisoning Against eBPF Runtime Security Monitors

## Abstract

Extended Berkeley Packet Filter (eBPF) has become the dominant instrumentation framework for cloud-native runtime security, with tools such as Falco, Tracee, and Tetragon deployed across millions of production nodes to detect container escapes, malicious process execution, and policy violations. These tools attach BPF programs to kernel tracepoints, kprobes, and LSM hooks, using BPF maps as the primary mechanism for configuration, state management, and inter-program communication. This paper identifies and empirically demonstrates a telemetry trust boundary weakness: **BPF maps lack per-map access control beyond the initial `CAP_BPF` capability check**, permitting any process with `CAP_BPF` to read, modify, or delete map entries belonging to any other BPF program on the same host — including security monitors.

We characterize **BPF Map State Poisoning**, a class of same-privilege tampering in which a process modifies the runtime BPF map state of a security tool to suppress event generation at the kernel level. We present three concrete instances: (1) modifying Tracee's `config_map` to zero `enabled_policies` and bump `policies_version`, causing `match_scope_filters()` to reject all events; (2) deleting Tetragon's `execve_calls` prog_array entries and clearing `execve_map`, breaking the tail-call-based execution pipeline and erasing all process tracking state; and (3) zeroing Falco's `interesting_syscalls` array, causing the BPF-side syscall filter to discard every system call before it reaches userspace.

Each modification was empirically verified against production releases (Tracee v0.24.1, Tetragon v1.4.0, Falco latest) and resulted in **complete telemetry suppression** — zero security events or alerts during active system activity that generated 14–16 events in unmodified baseline runs. The modifications require `CAP_BPF` (a privileged capability, but one that is realistically obtainable in post-exploitation and misconfigured container environments), take effect within one BPF program invocation, and produce no logs or alerts. We find that, to our knowledge, none of the tested tools employ `bpf_map_freeze()`, `BPF_F_RDONLY_PROG`, or any runtime integrity verification on their security-critical maps. We discuss defense-in-depth mitigations including map freezing, read-only program flags, BPF token scoping (kernel 6.9+), and userspace heartbeat mechanisms, and argue that this class of same-privilege tampering warrants attention from both tool maintainers and the broader eBPF security ecosystem.

**Keywords:** eBPF, BPF maps, map state integrity, runtime security monitoring, telemetry trust boundary, Falco, Tracee, Tetragon, cloud-native security, kernel security
